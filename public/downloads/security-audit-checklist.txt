AUTHENTICATION & SESSION MANAGEMENT
[ ] 1.  Passwords are hashed with bcrypt, Argon2, or scrypt (not MD5/SHA1)
[ ] 2.  Minimum password length enforced (12+ characters recommended)
[ ] 3.  Multi-factor authentication (MFA) available and encouraged for all accounts
[ ] 4.  Account lockout after 5-10 failed login attempts with exponential backoff
[ ] 5.  Session tokens are cryptographically random (128+ bits of entropy)
[ ] 6.  Sessions expire after inactivity (15-30 min for sensitive apps)
[ ] 7.  Session tokens invalidated on logout (server-side invalidation, not just cookie deletion)
[ ] 8.  "Remember me" tokens stored securely and rotated on use
[ ] 9.  Password reset links expire within 15-60 minutes and are single-use
[ ] 10. Login page protected against username enumeration (same response for valid/invalid email)

AUTHORIZATION & ACCESS CONTROL
[ ] 11. Every API endpoint checks authorization — never rely only on frontend hiding
[ ] 12. Horizontal privilege escalation tested: can user A access user B's resources?
[ ] 13. Admin routes require separate authorization check, not just "is logged in"
[ ] 14. Sensitive operations (delete, transfer, export) require re-authentication
[ ] 15. Role changes and permission grants are logged with who made the change and when
[ ] 16. Default deny — new users get minimum permissions, not maximum
[ ] 17. API keys have scopes — read-only keys cannot write

INJECTION ATTACKS
[ ] 18. All database queries use parameterized statements or an ORM — no string concatenation in SQL
[ ] 19. Input is validated at the API boundary (type, length, format) before processing
[ ] 20. HTML output is escaped to prevent XSS — innerHTML never set from user input
[ ] 21. File upload types validated server-side (not just client-side MIME type check)
[ ] 22. Uploaded files are not executed — stored outside webroot or in object storage
[ ] 23. XML parsing has external entity expansion (XXE) disabled
[ ] 24. LDAP, OS command, and template injection surfaces reviewed
[ ] 25. GraphQL introspection disabled in production; query depth and complexity limits set

HTTPS & TRANSPORT SECURITY
[ ] 26. HTTPS enforced on all endpoints — HTTP redirects to HTTPS (301, not 302)
[ ] 27. TLS 1.2 minimum; TLS 1.0 and 1.1 disabled
[ ] 28. HSTS header set with long max-age (minimum 1 year for prod)
[ ] 29. HSTS preload submitted if applicable
[ ] 30. Certificate auto-renews — expiry monitored with 30-day advance alert
[ ] 31. No mixed content warnings (HTTP resources on HTTPS pages)
[ ] 32. Cookies marked Secure, HttpOnly, and SameSite=Lax or Strict

SECURITY HEADERS
[ ] 33. Content-Security-Policy (CSP) set — blocks inline scripts and unknown origins
[ ] 34. X-Frame-Options: DENY or SAMEORIGIN (prevents clickjacking)
[ ] 35. X-Content-Type-Options: nosniff (prevents MIME sniffing)
[ ] 36. Referrer-Policy: strict-origin-when-cross-origin or stricter
[ ] 37. Permissions-Policy restricts camera, mic, geolocation if not needed

DEPENDENCY VULNERABILITIES
[ ] 38. npm audit / pip-audit / bundler-audit runs in CI on every PR
[ ] 39. Dependabot or Renovate enabled for automated dependency updates
[ ] 40. No packages with known Critical or High CVEs in production
[ ] 41. Unused packages removed from package.json / requirements.txt
[ ] 42. Lock files (package-lock.json, yarn.lock, Pipfile.lock) committed and used in builds

SECRETS MANAGEMENT
[ ] 43. No secrets in source code, git history, or client-side JavaScript
[ ] 44. `.env` files in .gitignore — confirmed by scanning git history
[ ] 45. Secrets rotated after any exposure — regardless of how brief
[ ] 46. API keys have minimum required permissions (principle of least privilege)
[ ] 47. Secrets stored in environment variable manager (Vercel, Doppler, AWS Secrets Manager) not in code

LOGGING & INCIDENT RESPONSE
[ ] 48. Authentication events logged: logins, logouts, failures, MFA events
[ ] 49. Authorization failures logged with user ID, resource, and timestamp
[ ] 50. Incident response plan exists: who to contact, how to revoke compromised credentials, how to notify affected users
